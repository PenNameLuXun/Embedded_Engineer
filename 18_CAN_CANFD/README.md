# 第 18 章　CAN / CAN-FD：汽车里最重要的协议

> CAN（Controller Area Network）是博世 1986 年为汽车设计的总线，今天每一辆车里都跑着几十条 CAN 网络。它的设计目标只有一个 —— **极端可靠**。这一章看看一个"为可靠而生"的协议长什么样。
>
> **学完本章你应该能**：(1) 解释 CAN 的差分 + 仲裁机制为什么能保证"最高优先级帧 0 延迟"，(2) 看懂一帧 CAN 报文，(3) 知道 CAN-FD 在标准 CAN 上加了什么，(4) 大致了解汽车 EE 架构里 CAN 在哪。

---

## 18.1 一句话定位

| 维度       | UART          | I²C            | SPI       | CAN              |
|------------|---------------|----------------|-----------|------------------|
| 异步/同步  | 异步          | 同步           | 同步      | 同步（差分自计时）|
| 主从       | 对等          | 主多从         | 单主多从  | **多主对等**     |
| 物理介质   | 单端          | 单端开漏       | 单端      | **差分双绞**     |
| 最大长度   | 米            | 米             | 米        | **40 m @ 1 Mbps，km @ 50 kbps** |
| 仲裁       | 无            | 有             | 无        | **逐位非破坏性仲裁** |
| 错误处理   | 软件          | 软件           | 软件      | **协议内置 + 自动重发** |

CAN 的杀手锏是 **"非破坏性仲裁"** —— 多个节点同时发，最高优先级的一个继续传不受影响。

---

## 18.2 物理层：差分双绞

```
        VCC
         │
   ──────┴──────  CAN_H  ──┐
                            ├── 双绞线
   ──────┬──────  CAN_L  ──┘
        GND

  发"0"（dominant）：CAN_H 高，CAN_L 低（差分 ~2 V）
  发"1"（recessive）：两根都中点（差分 ~0 V，靠终端电阻保持）
```

两个 120 Ω 终端电阻接总线两端。**"显性 0 覆盖隐性 1"** —— 这是仲裁能工作的物理基础（电气线与，但反极性）。

---

## 18.3 帧格式（经典 CAN，11 位 ID 标准帧）

```
SOF | Arbitration |  Control  |  Data 0..8 B | CRC | ACK | EOF | IFS
 1b      12 b          6 b        0..64 b      16b   2b    7b    3b

  └──── ID 11 b + RTR 1 b ─────┘└─ IDE+r0+DLC ──┘
```

**字段**：
- **SOF (Start Of Frame)**：1 位显性
- **ID + RTR**：标识符 11 位 + 远程帧位
- **Control**：IDE (扩展帧标志) + r0 + DLC (数据长度 0-8 字节)
- **Data**：0–8 字节
- **CRC**：15 位 CRC + 1 位定界符
- **ACK**：发送方发隐性，任何收到完整正确帧的节点拉显性 → 发送方知道至少一个节点收到了

**关键事实**：
- 报文不带地址字段，只带 ID = **"内容标识"**。任何节点发"温度=25"，所有需要温度的节点都接收。这叫"基于内容的总线"。
- ID 越小优先级越高（显性 0 在前）
- 扩展帧用 29 位 ID（标准 + 扩展 18 位）

---

## 18.4 非破坏性仲裁

多个节点同时 SOF，开始发各自 ID。**逐位比较**：

```
节点 A: ID = 0x100  =  0001_0000_0000
节点 B: ID = 0x120  =  0001_0010_0000
                            ↑
                       此时 A 发 0, B 发 1
                       线上为 0 (显性赢)
                       B 看到自己发 1 但线是 0 → 退出仲裁
                       A 继续完整发完
```

**精髓**：
- A 的报文不被破坏，照常完成
- B 等总线空闲后重发
- ID 静态分配，关键报文给小 ID → **天然 priority queue**

汽车里"刹车控制"ID 是个位数；"车窗状态"ID 几百。永远刹车赢。

## 18.5 错误检测与处理（协议级）

CAN 内置 **五种检错**：

| 错误         | 检测方式                            |
|--------------|-------------------------------------|
| CRC error    | 接收方算 CRC 不匹配                 |
| ACK error    | 发送方没收到 ACK                    |
| Form error   | 帧固定位置错（应该显性的位置隐性）  |
| Bit error    | 发送方监听自己发的位与线上不符      |
| Stuff error  | 5 个相同位后未插 stuff 位           |

> **位填充 (Bit Stuffing)**：每 5 个相同位强制插一个反极性位，保证有边沿让接收方再同步。这是 CAN 的"自时钟"机制。

任何错都触发**错误帧** → 总线进入错误恢复 → 发送方自动重发。**协议级别保证"成功送达"**。

### 错误状态机

每个节点维护两个计数器：TEC (发送错误计数) 和 REC (接收错误计数)：

```
ERROR ACTIVE  (TEC<128)：正常发送，错误后能发主动错误帧
       ↓ TEC 增加
ERROR PASSIVE (128≤TEC<256)：还能发，但只发被动错误帧
       ↓ TEC 增加
BUS OFF       (TEC≥256)：断开总线 (硬件下线)
```

故障节点逐渐"边缘化"再"下线"，防止它把整条总线搞瘫。

## 18.6 CAN-FD：加宽 + 提速

经典 CAN 极限 1 Mbit/s，数据最多 8 字节。CAN-FD（**Flexible Data-rate**）改进：
- 数据段最多 **64 字节**
- 数据段可切换到更高比特率（仲裁段还是 1 Mbps，数据段可到 5–8 Mbps）
- 帧结构兼容（在 IDE/EDL 位区分）

为什么数据段能提速？仲裁段需要电气线与，距离决定最大波特率。数据段只有发送方一家在发，可以飞起来。

## 18.7 CAN 控制器 IP 核内部

```
┌──────────────────────────────────────────────────────────┐
│                                                            │
│   ┌──────┐    ┌──────────────────┐    ┌─────────────┐      │
│   │ 总线 │    │   TX 邮箱        │    │             │      │
│   │ 接口 │───→│  (TX FIFO)       │───→│  比特处理   │      │
│   │      │    └──────────────────┘    │   单元      │      │
│   │      │    ┌──────────────────┐    │ (BSP)       │      │
│   │      │←───│   RX 邮箱        │←───│             │      │
│   │      │    │ + 接受滤波器     │    │             │      │
│   │      │    └──────────────────┘    └─────────────┘      │
│   │      │    ┌──────────────────┐         ↓               │
│   │      │←───│ 错误计数 / 状态  │    ┌─────────────┐      │
│   │      │    └──────────────────┘    │ 位时序 +    │      │
│   └──────┘                            │ 同步       │       │
│                                       │ + Stuff    │       │
│                                       └─────────────┘      │
│                                            ↓               │
│                                       CAN_TX / CAN_RX     │
│                                       (接 PHY 转 CAN_H/L)  │
└──────────────────────────────────────────────────────────┘
```

**比特处理单元 (BSP, Bit Stream Processor)**：
- 串行化 / 反串行化
- 位填充 / 反填充
- CRC 计算
- 错误检测

**位时序单元**：
- 一个 bit 分成 SyncSeg + PropSeg + PhaseSeg1 + PhaseSeg2 = 8–25 个时间份额 (TQ)
- 采样点通常在 75%–87.5%
- 边沿到来时动态调整 PhaseSeg 长度 → 重同步

这一段细节决定 CAN 总线长度和波特率的关系。

## 18.8 汽车 EE 架构里的 CAN

```
┌───────────────────────────────┐
│       域控 / 网关 ECU         │
└────┬──────┬──────┬──────┬─────┘
     │      │      │      │
   CAN-A  CAN-B  CAN-C  CAN-D
   动力   底盘   车身   信息娱乐
     │      │      │      │
  发动机  转向   车窗    收音
  变速箱  刹车   座椅    导航
   ...    ...    ...     ...
```

不同总线运行在不同速率：
- 动力 / 底盘：500 kbps – 1 Mbps，关键
- 车身 / 舒适：100 kbps – 500 kbps
- 诊断 (UDS over CAN)：500 kbps

逐步被 **CAN-FD / Automotive Ethernet (100/1000Base-T1)** 替代，但 CAN 在低端 ECU 还会活很久。

## 18.9 QEMU + 软件 CAN 体验

QEMU 不模拟车载 CAN 控制器，但 **Linux 提供 SocketCAN**：

```bash
# 在主机 Linux 上（不需要硬件）
sudo modprobe vcan
sudo ip link add dev vcan0 type vcan
sudo ip link set up vcan0

# 发一帧
cansend vcan0 123#DEADBEEF

# 收
candump vcan0
```

这是真实工程师调 CAN 的常用环境。`code/socketcan_demo/` 给出一个简单的 C 程序 send + recv。
